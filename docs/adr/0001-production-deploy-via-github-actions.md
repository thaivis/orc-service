# Production deploys run as a push-based GitHub Action, not GitOps

Production deploys were previously fully manual: a human hand-edited the image tag in `k8s/deployment.yaml` and ran `kubectl apply`/`rollout status` locally. We added a `workflow_dispatch` GitHub Action (`.github/workflows/deploy.yml`) that runs on a self-hosted runner with `production` cluster access already configured, applies the tag bump, waits for rollout, and — only on a successful rollout — commits the change back to `main`. We chose this push-based model (Action pushes to the cluster) over a pull-based GitOps controller (e.g. Flux/ArgoCD watching the repo) because no such controller exists in the cluster today and introducing one is a bigger infrastructure change than this deploy action warrants. The Action only ever applies `k8s/deployment.yaml`; the other split-out resources (`configmap.yaml`, `service.yaml`, `hpa.yaml`, `networkpolicy.yaml`) still require a manual `kubectl apply -f k8s/`, same as before.

Commit-after-rollout (rather than commit-then-apply) was a deliberate choice: it keeps `main` truthful about what's actually running in production, at the cost of a brief window where the cluster runs an image not yet reflected in git.
